# 项目错题集

用途：只记录本项目协作中已经真实发生过的错误，以及对应的复盘和防复发流程。

不记录泛化建议、预防性规则、项目原则或尚未真实发生的假设案例。那些内容应放在 `AGENTS.md`、`README_网站维护.md`、`小哲摄影师网站建设基石.md` 或具体专题文档里。

## 使用方式

每次遇到类似任务时，先检查这里是否有已经发生过的同类错误。

如果有，必须先复用已验证路径，再动手修改。只有在明确复用失败并说明具体阻断点后，才请求用户补材料。

## 真实错题

### 错题卡 1：没有复用已经跑通过的工作流

- 发生场景：Google 评价模块更新。
- 真实触发：用户说又多了一条 Google 评价，并提供了客户授权头像，要求首页和 Contact 页都做出来。
- 错误表现：此前已经跑通过“登录 Google → 打开商家资料 → 读取评价 → 提取客户名和原文 → 同步首页和 Contact 页”的路径；第二次处理时没有先复原这条路径，而是把任务简化成“用户给了头像，所以让用户再补评价原文”。
- 直接后果：页面先出现了 `待补客户姓名 / 待补真实 Google 评价` 这类占位内容，把本该由我查证完成的工作退回给用户。

#### 错误根因

1. **任务识别失败。** 没有把这次识别为“复用已跑通过流程”，而是当成新的页面占位任务。
2. **注意力偏移。** 过度关注用户新提供的头像，弱化了“Google Business Profile 里已经有评价”这个事实来源。
3. **没有回溯成功路径。** 即使上下文压缩导致细节丢失，也应该从当前浏览器、本地文件、已有改动和项目文档中反向复原。
4. **执行上选择了低阻力路径。** 查 Google 登录态和商家面板比写占位麻烦，但用户已经给了足够线索，不能把查证责任推回给用户。

#### 正确处理方式

```text
先判断这是否是已跑通过的同类任务。若是，必须优先复用旧路径：打开当前已登录 Google 页面或商家资料面板，查真实评价、客户名、评分和原文；只有在明确读不到时，才说明阻断点，并请求用户补充。
```

#### 防复发流程

1. 遇到“又多了一条”“和上次一样”“继续之前那个”“你上次又忘了”“你自己去找”“两页都要”这类表达，先判断是否已有成功先例。
2. 如果已有成功先例，先复原成功路径，不直接做占位。
3. 如果上下文压缩导致记忆不完整，明确说明“我需要先复原上次路径”，然后查浏览器状态、文件 diff、已有素材和项目文档。
4. 请求用户补材料前，必须说明已经尝试过哪些路径、具体卡在哪里。
5. 涉及网站多处同步时，必须检查正式发布目录下的 `public/zh/index.html` 和 `public/zh/contact.html` 是否都改到；根目录同名 HTML 不再作为上线源。

#### Google 评价复用路径

1. 读取本文件、`README_网站维护.md` 的 Google 评价模块维护标准，并查看 `public/zh/index.html`、`public/zh/contact.html`、`public/styles.css` 中当前评价模块结构。
2. 检查当前浏览器是否已经登录 Google，并优先使用用户当前 Google 搜索页或商家面板。
3. 如果 Business Profile 后台 locations 页面显示 0 个商家，不立即放弃，继续从 Google 搜索商家面板进入“查看商家资料 / 阅读评价 / 评价”。
4. 提取客户名、评分、原文、时间和来源；若 Google 显示翻译，优先寻找“查看原文”或页面中隐藏的英文原文。
5. 同步修改 `public/zh/index.html` 和 `public/zh/contact.html`。
6. 修改后检查占位文案是否清空、头像是否存在、两页文字是否一致。

#### 禁止做法

- 没有先查 Google，就让用户自己复制评价原文。
- 用“待补真实评价”替代已经能查到的真实评价。
- 只改首页或只改 Contact。
- 把 Google 翻译文本当成客户英文原文，除非确认找不到原文并明确标注。

#### 输出前自检

- 这件事是否已有成功先例？
- 是否先复用了成功路径？
- 如果没有复用，是否说明了客观阻断原因？
- `public/zh/index.html` 和 `public/zh/contact.html` 是否都同步？
- 是否把查证责任不必要地推回给用户？

### 错题卡 2：漏审正式页面中的内部提醒与策略文案

- 发生场景：隐私政策页、首页和流程页上线前检查。
- 真实触发：用户把网站交给外部 ChatGPT 审查后，对方先指出 `public/zh/privacy.html` 中仍公开显示“上线前提醒”，包含确认商业名称、法律地址、数据分析设置、浏览器记录工具和表单服务设置等内部 TODO；随后又指出 `public/zh/index.html` 中出现“首页先让客户感受到...”和“把解释留给服务模块...”这类内部策略型文案。2026-06-15，用户又在 `public/zh/process.html` 指出“为什么流程页重要 / 很多客户不是不想咨询...”这类解释模块价值的文案像是给内部看的，而不是给客户看的。
- 错误表现：此前自查时只关注了首页、Booking、FAQ、Contact 等主要转化路径是否可用，没有把所有公开页面和首页所有文案都按客户视角逐句审查，导致内部提醒、给协作者看的策略说明和解释模块存在理由的内部论证句混入正式发布目录。
- 直接后果：客户如果看到这些内容，会感觉网站没有完成，或觉得页面在解释设计策略而不是和客户沟通，影响高级感、专业感和商业可信度。

#### 错误根因

1. **审查范围偏窄。** 把上线检查重点放在转化页面，没有逐页检查所有 `public/zh/` 下的正式页面。
2. **把隐私页当成辅助页面。** 没有按公开客户页面标准检查它的文案完成度、法律表述和内部 TODO。
3. **客户视角审查缺失。** 没有逐句判断文案到底是在对客户说话，还是在对设计师、Codex 或项目协作者解释页面策略、模块价值或转化逻辑。
4. **关键词检查缺失。** 没有在正式发布目录中搜索“上线前、正式发布前、待确认、待补、TODO、占位、首页先、服务模块、第一阶段隐藏、草稿模块、为什么、重要、很多客户、降低犹豫”等风险词。
5. **视觉检查不完整。** 没有逐页打开正式页面确认客户实际会看到什么。

#### 正确处理方式

```text
上线前审查必须覆盖 public/zh/ 下所有正式页面，包括 privacy.html 和 process.html。每次检查都要搜索内部提醒、占位关键词、内部策略型表达和解释模块价值的句子，并打开页面确认可见内容没有暴露内部 TODO、给协作者看的页面说明或内部转化逻辑。
```

#### 防复发流程

1. 涉及上线安全、正式发布或整站审查时，先列出 `public/zh/` 下全部正式页面。
2. 对正式发布目录执行关键词检查：`上线前`、`正式发布前`、`待确认`、`待补`、`TODO`、`占位`、`首页先`、`服务模块`、`第一阶段隐藏`、`草稿模块`、`为什么`、`重要`、`很多客户`、`降低犹豫`。
3. 对 Privacy、FAQ、Booking、Contact 这类含事实或法律/商业承诺的页面，按公开客户页面标准检查，不把它们当作低优先级附属页。
4. 对首页、Services、Portfolio、About、Process 这类营销和转化辅助页面，逐句检查是否是客户视角表达；凡是解释页面结构、模块职责、阶段规划、内容策略或模块为什么存在的句子，应移到项目文档，不出现在正式页面。
5. 若页面必须保留未确认事项，只能放在项目文档，不允许出现在正式页面可见文案里。
6. 修改后至少检查桌面和手机宽度，确认没有横向溢出、内部提醒残留、内部策略文案残留和导航不可用问题。

#### 输出前自检

- 是否逐页检查了 `public/zh/` 下的正式页面？
- 是否搜索过内部提醒、占位关键词和内部策略型表达？
- Privacy 页是否像正式公开页面，而不是内部 TODO 页面？
- 首页、流程页和其他营销页面文案是否是在对客户说话，而不是解释页面策略、模块价值或转化逻辑？
- 是否把法律、隐私、表单、数据保留相关表述写成客户可理解的正式说明？
- 是否避免把未确认法律地址、后台设置、内部操作项暴露给客户？

### 错题卡 3：只验证 JS 筛选，漏验跨页 hash 锚点

- 发生场景：Booking 页面跳转到 Portfolio 分类作品。
- 真实触发：外部 ChatGPT 审查指出 `public/zh/booking.html` 中的 `/zh/portfolio.html#portrait`、`#couple`、`#wedding-elopement` 链接，在 `public/zh/portfolio.html` 中没有对应 `id` 锚点。2026-06-15，用户又指出 `public/zh/faq.html#planning` 这类 FAQ 目录锚点虽然能跳转，但目标标题会被固定页头遮挡。同日用户再次点击 FAQ 左侧“预订与价格”后截图指出，`#booking` 标题仍被页头遮挡，说明前一次修复只加了 CSS 留白，没有充分验证真实点击路径。2026-06-16，用户第三次截图指出点击 FAQ 左侧“拍摄当天”等目录后，页面仍停在错误位置或标题没有按预期露出，说明修复还没有处理平滑滚动时序、浏览器 hash 行为和资源缓存问题。
- 错误表现：此前只看到 `public/script.js` 会读取 hash 并应用作品筛选，就以为跳转路径可用；后续又只补了 `scroll-margin-top`，没有检查 HTML 层面是否存在真实锚点，也没有验证 JS 禁用、加载失败、浏览器原生跳转时的语义兜底、固定页头下的滚动落点是否可见，以及用户从页面目录实际点击后的最终位置。
- 直接后果：服务与报价页到作品页、FAQ 目录到分类标题这类关键路径不够稳。JS 正常时筛选可用，但 HTML 本身没有落点时浏览器不能可靠滚动到对应筛选区；有真实锚点但没有滚动留白时，客户会跳到被页头遮住的位置。

#### 错误根因

1. **过度信任 JS 状态。** 只确认脚本能根据 hash 筛选，没有检查 URL 片段是否对应真实 DOM `id`。
2. **跨页和页内链路 QA 不完整。** 没有从 Booking 的“查看作品”链接、FAQ 左侧目录等入口出发，实际打开目标 URL 验证落点、筛选状态、滚动位置和固定页头遮挡。
3. **无 JS 兜底意识不足。** 静态网站的关键链接应先在 HTML 层可解释，再由 JS 增强体验。
4. **转化路径优先级判断不足。** Booking 到 Portfolio 是客户决策路径，不应只按普通内部链接检查。
5. **修复后验证对象错误。** 修复 FAQ 锚点时，只确认 CSS 规则存在或直接打开 URL 的效果，没有复现用户的真实操作：在页面内点击左侧目录，再观察右侧标题是否完整露出。
6. **缓存与滚动时序意识不足。** 修改共享 CSS / JS 后没有给 FAQ 页面资源加版本号，也没有关闭平滑滚动和重复校正，导致用户当前浏览器可能继续使用旧脚本，或在平滑滚动未结束时看到错误落点。

#### 正确处理方式

```text
凡是正式页面出现 #hash 链接，必须同时检查目标页面存在对应 id、JS 增强逻辑能正确响应、无 JS 时链接仍有基本语义，并确认固定页头不会遮挡目标标题；关键转化路径和目录跳转要用浏览器实际打开验证，也要从页面上的真实入口点击验证。
```

#### 防复发流程

1. 搜索正式目录中的 `href`，尤其是包含 `#` 的跨页链接。
2. 对每个 `/path#id`，检查目标页面是否存在唯一的 `id="id"`。
3. 如果 hash 同时驱动 JS 状态，例如筛选、展开、滚动，应验证 JS 正常时的激活状态和显示结果。
4. 对 Booking、Contact、Portfolio、FAQ 等关键转化路径，必须用浏览器打开实际 URL，确认落点、滚动位置、目标标题是否被固定页头遮挡，以及手机端无横向溢出。
5. 对会被固定页头覆盖的目标元素，补 `scroll-margin-top` 或全局 `scroll-padding-top`，让浏览器原生锚点跳转留出页头空间。
6. 对 FAQ 这类页内目录，修复后必须从左侧目录实际点击每个 hash，记录 `h2` 顶部是否低于页头底部并留有安全距离；不能只直接打开 `URL#hash`。
7. 如果某个 hash 只由 JS 解释，应补真实锚点或改成更明确的 URL 方案，不能只依赖脚本。
8. 修改 hash 锚点相关的共享 CSS / JS 后，应确认页面加载了新资源；必要时给目标页面的样式或脚本链接加版本号，并用浏览器坐标验证滚动结束后的最终位置。

#### 输出前自检

- Booking 到 Portfolio 的所有分类链接是否有真实目标 `id`？
- 点击或直接打开 hash URL 后，筛选按钮是否正确激活？
- 可见作品是否符合当前分类？
- 浏览器是否滚到客户能理解的位置？
- 目标标题是否没有被固定页头遮挡？
- 是否从页面内真实目录或 CTA 点击过，而不是只直接打开目标 URL？
- JS 禁用或失败时，链接是否至少能落到相关区域？

### 错题卡 4：只看页头一致，漏审页脚导航一致性

- 发生场景：正式中文页面导航一致性检查。
- 真实触发：外部 ChatGPT 审查指出 `public/zh/*.html` 页脚链接会按当前页面变化，例如首页页脚缺少“首页”，Booking 页脚缺少“服务与报价”，Contact 页脚缺少“联系咨询”。
- 错误表现：此前检查时确认了页头导航一致，也确认了主 CTA 和 Contact 路径可用，但没有把页脚作为同等级的重复组件逐页对比；页脚实际采用了“省略当前页链接”的手工写法，导致每页 footer 链接集合不一致。
- 直接后果：这类差异短期不影响客户完成咨询，但会增加后续维护成本。未来新增多语言、SEO 链接或统一导航调整时，容易只改到部分页面。

#### 错误根因

1. **把页脚当作低风险区域。** 检查重点放在页头、表单、Portfolio 筛选和主要 CTA，页脚只被当成辅助链接区。
2. **没有做组件级对比。** 静态站没有共享模板，重复写在每个 HTML 文件里的页脚必须按组件清单比对，不能只看单页是否能用。
3. **误把“省略当前页”当成可接受设计。** 对单页浏览来说这看起来合理，但对长期维护和多语言复制来说会制造差异。
4. **没有复用项目规则。** `AGENTS.md` 已要求修改页头导航、页脚、主 CTA、Contact 链接时检查主要页面一致性，但执行时只覆盖了部分区域。

#### 正确处理方式

```text
凡是静态站里重复出现在所有页面的区域，包括页头、页脚、主 CTA 和 Contact 入口，都要按同一组链接、同一顺序、同一目标逐页比对。当前页状态应通过 class 或脚本标记，不通过删除当前页链接实现。
```

#### 防复发流程

1. 涉及导航、页脚、CTA、Contact 入口或多语言结构时，先列出 `public/zh/` 下全部正式页面。
2. 分别抽取页头导航和页脚链接，比较链接文字、顺序、href 和数量。
3. 统一页脚时保留完整链接集合；当前页只加 `is-active` 或同类状态，不删除链接。
4. 修改共享导航区域后，用浏览器检查桌面和手机宽度，确认菜单、页脚 active、Contact 链接和 Portfolio 筛选没有受影响。
5. 后续新增语言目录时，先复制已统一的中文结构，再翻译和本地化，不从旧稿或不一致页面复制。

#### 输出前自检

- `public/zh/` 下所有正式页面页脚链接数量是否一致？
- 页脚链接顺序和目标是否一致？
- 当前页是否通过 active 状态标记，而不是删除链接？
- 页头导航、页脚、主 CTA 和 Contact 链接是否都检查过？
- 手机菜单、Portfolio 筛选和 Contact 表单是否仍可用？

### 错题卡 5：只把 Portfolio 当作品浏览页，漏掉首屏附近转化入口

- 发生场景：Portfolio 精选作品页转化路径检查。
- 真实触发：外部 ChatGPT 审查指出 `public/zh/portfolio.html` 的主要 CTA 位于页面底部，顶部只有作品筛选；客户看到喜欢的图时，需要滑到底部才有明确咨询入口。
- 错误表现：此前把 Portfolio 的重点放在作品展示、分类筛选、hash 锚点和视觉流畅度，没有按“强转化页”标准检查首屏附近是否有轻量咨询入口。
- 直接后果：作品页能建立审美，但客户在浏览前半段缺少明确下一步，和项目基石中“客户看完知道下一步点哪里”的要求不完全一致。

#### 错误根因

1. **页面角色判断不完整。** 把 Portfolio 当成作品集合，而没有同时当成客户决策页。
2. **只检查了底部 CTA 是否存在。** 没有判断 CTA 是否出现在客户做出兴趣判断的关键位置。
3. **筛选功能被当作顶部主要动作。** 筛选能帮助浏览，但不能替代咨询入口。
4. **项目基石执行不完整。** 已有规则要求页面服务转化、让客户知道下一步，但检查时没有把这个要求落实到 Portfolio 顶部区域。

#### 正确处理方式

```text
Portfolio 这类强审美页面必须同时检查作品展示和咨询路径。顶部或筛选区附近应有轻量 CTA，让客户在产生兴趣时能立即咨询类似拍摄；底部 CTA 只能作为二次收束，不应是唯一入口。
```

#### 防复发流程

1. 检查作品页时，同时检查首屏、筛选区、作品流中段和底部 CTA。
2. 若页面承担客户决策作用，筛选、作品或介绍模块附近至少要有一个明确 Contact / Inquiry 入口。
3. CTA 文案应连接客户当下意图，例如“喜欢这种风格？咨询类似拍摄”，不要只写泛泛的“联系我们”。
4. 修改后在手机宽度下检查 CTA 是否可见、不会挤压筛选按钮、不会造成横向溢出。
5. Portfolio 改动后继续验证筛选、hash 锚点、页头菜单、页脚链接和 Contact 表单状态。

#### 输出前自检

- Portfolio 首屏或筛选区附近是否有咨询入口？
- 客户看到喜欢的图后，是否不需要滑到底就能咨询？
- CTA 是否指向正式 Contact 页？
- CTA 文案是否和作品浏览场景相关？
- 手机端筛选和 CTA 是否都清晰可用？

### 错题卡 6：Contact 预算档位没有跟 Booking 起价同步

- 发生场景：Contact 表单预算选项与 Booking / Pricing 页面价格口径检查。
- 真实触发：用户指出 `public/zh/contact.html` 的预算下拉里有“300 欧以下”，但 `public/zh/booking.html` 已经公开写明最低服务为“个人 / 旅行肖像 299 欧起”。
- 错误表现：此前修改和审查 Booking 价格、Contact 表单时，只分别确认了页面自身可用，没有把表单预算档位与正式价格起点放在一起校验。
- 直接后果：客户可能选择低于实际起价的预算范围，造成价格预期错误；也会让网站内部价格体系显得不一致。

#### 错误根因

1. **把表单选项当成低风险文案。** 预算下拉看起来只是咨询辅助字段，但它实际会影响客户对最低价格的理解。
2. **跨页面价格口径检查不足。** 只检查 Booking 是否有价格，没有同步检查 Contact、FAQ、Process 中与预算、订金、报价相关的入口。
3. **没有建立价格变更联动清单。** 一旦 Booking 起价确认，Contact 表单、咨询提示和 FAQ 的相关表达都应同步核对。

#### 正确处理方式

```text
凡是修改或确认 Booking 起价、订金、交付周期、加时费、异地费，都必须同步检查 Contact 表单选项、FAQ、Process 和首页 CTA 附近的价格/预算表达，避免一个页面公开起价，另一个页面继续保留更低或旧口径。
```

#### 防复发流程

1. 搜索正式目录中的价格与预算关键词：`欧`、`预算`、`报价`、`起`、`订金`、`加时`、`异地`。
2. 以 `public/zh/booking.html` 的正式价格为主口径，核对 Contact 表单预算档位是否低于最低起价。
3. 如果价格只是起价，Contact 选项用“大致范围”表达，不写低于起价的档位。
4. 修改后提交一次表单前端检查，确认预算字段值、成功/失败状态和必填字段不受影响。

#### 输出前自检

- Contact 是否还存在低于 Booking 起价的预算选项？
- Booking、FAQ、Process、Contact 的价格和预算表达是否互相矛盾？
- 表单提交时预算字段是否仍能正常带出选择值？
- 修改是否只发生在正式发布目录 `public/`？

### 错题卡 7：把摄影师本人照片夹在首页作品流里

- 发生场景：首页精选影像 / 滚动照片流模块。
- 真实触发：用户在浏览器中框选首页中段 `关于 Fiona` 模块，指出它下方已经有正式 About 模块，而且这张 Fiona 工作照夹在作品流里，客户不细看文案时容易误以为这是客片。
- 错误表现：此前把摄影师本人介绍卡和本人工作照放进首页作品流，试图在作品流中穿插人格信任信息，但没有充分区分“摄影师本人照片”和“客户作品照片”的视觉身份。
- 直接后果：客户可能误判照片主体身份，影响作品真实性和商业可信度；同时 About 信息在首页重复出现，降低页面节奏。

#### 错误根因

1. **素材身份判断不足。** 只考虑这张照片能补充 Fiona 的真实感，没有判断它出现在作品流中会不会被当成客片。
2. **模块职责重叠。** 首页已有正式 About 模块，作品流中再插入 `关于 Fiona` 会造成信息重复。
3. **客户快速浏览场景考虑不足。** 客户浏览图片时未必逐字读文案，不能依赖小字说明来区分摄影师本人和客户作品。

#### 正确处理方式

```text
首页作品流、Portfolio 和服务页样片应默认只放客户作品或明确的风格样片。摄影师本人照片应放在 About、联系页或明确标注的个人介绍模块里，不夹在作品流中。
```

#### 防复发流程

1. 修改首页、Portfolio、服务页照片流前，先判断每张照片的主体身份：客户作品、摄影师本人、场地细节、氛围图。
2. 客户作品流中不要混入摄影师本人照片；确需展示摄影师本人时，用 About 模块、独立标题、figcaption 或清楚上下文隔开。
3. 检查首页是否重复表达同一件事：如果已有正式 About 模块，作品流里不再放同功能的 `关于 Fiona` 卡片。
4. 浏览器检查时按客户快速扫图方式看页面，不只按代码语义判断。

#### 输出前自检

- 首页作品流里是否只保留客户作品或明确风格样片？
- 摄影师本人照片是否只出现在 About 或明确个人介绍区域？
- 是否存在客户可能误以为本人照片是客片的布局？
- 同一页是否重复出现功能相近的 About 模块？

### 错题卡 8：把明确的视觉位置要求当成模糊间距调整

- 发生场景：Portfolio 筛选区附近的轻量咨询 CTA。
- 真实触发：用户多次指出 `public/zh/portfolio.html` 中“喜欢这种风格？带上日期和地点 / 询问档期”这句话需要上移到筛选按钮和作品图片之间的正中间，并要求“询问档期”做成红色、带下划线的参考图样式。2026-06-16，用户第三次截图指出这句话仍然贴近图片，且“带上日期和地点”后仍有句号。
- 错误表现：前两次修改只凭视觉感觉调了 `gap`、`margin-bottom` 和链接样式，没有把用户说的“正中间”转成可验证的几何标准，也没有确认最终页面文字是否去掉句号。
- 直接后果：同一个局部视觉问题反复返工，用户需要重复解释同一句要求，降低对实现准确性的信任。

#### 错误根因

1. **没有把自然语言转成坐标约束。** 用户说的“正中间”应被拆成：筛选按钮底部到 CTA 顶部的距离，等于 CTA 底部到图片顶部的距离。
2. **验收标准错误。** 只看截图觉得“差不多”，没有量 `filter-bar`、`portfolio-quick-cta` 和 `portfolio-grid` 的真实位置。
3. **细节清单遗漏。** 用户明确要求删除“带上日期和地点”后的句号，但前一轮只关注链接样式和大概位置，漏掉文字内容。
4. **样式表达不够直接。** 对“红色字体 + 下划线”使用了较隐性的伪元素方案，虽然技术上可见，但不如直接设置红色和 `border-bottom` 稳定清楚。

#### 正确处理方式

```text
对于“居中、对齐、上移、下移、和参考图一样”这类具体视觉要求，必须先定义可测量标准，再改样式。修改后用浏览器读取目标元素坐标，确认差值为 0 或在 1px 以内，同时检查文案字符是否完全符合用户要求。
```

#### 防复发流程

1. 遇到“正中间”“对齐”“上移”“下移”时，先确定参照物：哪个元素的顶部、底部、中心线或边缘。
2. 对涉及间距的需求，修改后用浏览器读取 `getBoundingClientRect()`，输出上方距离、下方距离和中心偏差。
3. 对涉及参考图样式的需求，检查颜色、字体、下划线、字号和可点击范围，不只看类名是否存在。
4. 对涉及文字的小改动，单独检查最终 `innerText`，确认标点、换行和链接文字准确。
5. 如果用户已经第二次指出同一问题，不能再用主观截图验收，必须给出坐标或像素级验证结果。

#### 输出前自检

- 是否明确了参照元素？
- 是否量过上下距离或左右对齐差值？
- 是否确认用户指定的标点已经删除或保留？
- 参考图里的关键样式是否真的可见？
- 是否在桌面和手机宽度都检查过？

### 错题卡 9：只验证 GitHub 默认地址，漏确认正式域名

- 发生场景：中文版静态站第一次发布到 GitHub Pages。
- 真实触发：用户看到页面地址是 `arno23333.github.io/zh/`，追问为什么不是自己在 Google / Squarespace 买的正式域名，并要求推送到那个网址。
- 错误表现：最初把 GitHub Pages 默认地址能打开当作上线完成，没有先确认客户最终要对外使用的域名、DNS 托管平台、邮箱记录和 HTTPS 证书状态。
- 直接后果：网站内容虽然已经部署，但正式域名、DNS 指向和 HTTPS 证书还没完成，用户在浏览器中继续看到证书错误和 GitHub 默认地址，造成“上线了但不像正式上线”的结果。

#### 错误根因

1. **上线定义不完整。** 只验证托管平台默认地址，没有把客户购买的正式域名纳入验收条件。
2. **DNS 责任边界没有提前确认。** 没有在发布前确认域名注册商、DNS 托管方、邮箱记录和网站记录各自在哪里维护。
3. **HTTPS 等待期没有提前解释。** GitHub Pages 自定义域名绑定后，DNS Check 和证书签发可能需要等待；没有提前把“HTTP 可用”和“HTTPS 正式可用”分开说明。

#### 正确处理方式

```text
网站上线任务不能只以 GitHub Pages 默认地址可访问为完成。必须确认最终对外域名、DNS 指向、HTTPS 证书、HTTP 到 HTTPS 跳转和正式页面路径都可用，才算正式上线完成。
```

#### 防复发流程

1. 上线前先问或查清最终对外域名，不默认用托管平台临时域名。
2. 检查域名 DNS 托管位置，区分网站记录和邮箱记录；修改 DNS 时保留 Google Workspace / 邮箱相关 MX、SPF、DKIM 记录。
3. GitHub Pages 使用自定义域名时，确认 `public/CNAME` 与 GitHub Pages Settings 中的 Custom domain 一致。
4. DNS 修改后分别验证：`www` CNAME、根域名 A 记录、HTTP 状态、HTTPS 状态、证书 Subject、GitHub Actions 最新部署。
5. 如果 HTTPS 证书尚未签发，明确告诉用户这是等待 GitHub DNS Check / 证书签发，不要误判为页面未上线。

#### 输出前自检

- 最终对外链接是否是客户购买的正式域名？
- 是否验证了 `https://www.fionatangstudio.com/zh/`，而不只是 `https://arno23333.github.io/zh/`？
- HTTP 是否自动跳转 HTTPS？
- 证书 Subject 是否为 `www.fionatangstudio.com`？
- DNS 修改是否保留了邮箱记录？

### 错题卡 10：只验图片加载成功，漏验作品是否真实可见

- 发生场景：SEO 和图片结构优化后，Portfolio 作品页上线检查。
- 真实触发：2026-06-19，用户指出线上正式站和本地预览的 Portfolio 作品照片都不可见；截图显示筛选区下方大面积空白。
- 错误表现：SEO 提交中给 `.image-tile picture` 增加了全局 `display / width / height / position` 样式，和 Portfolio 瀑布流中 `.website-masonry-item img { position: static; }` 的规则冲突。图片资源本身能加载，`img.naturalWidth` 正常，但 `picture` 仍处于绝对定位，作品卡片父级高度被压成 `0`，再被 `overflow: hidden` 裁掉，客户实际看到的是空白。
- 直接后果：Portfolio 作为强转化页面失去核心作品展示，线上客户进入后无法判断摄影风格；后续审核仍然没有发现，说明审核标准没有覆盖真实可见性。

#### 错误根因

1. **把资源加载当成视觉可见。** 只确认图片地址、HTTP 状态或 `naturalWidth`，没有检查图片和作品卡片的真实渲染高度。
2. **低估 SEO / 图片结构改动的视觉风险。** 给 `picture`、响应式图片、懒加载、尺寸属性或共享图片类名加规则时，会影响所有使用同一组件的页面，不能只按 SEO 元信息检查。
3. **没有从客户视角走到关键内容区。** 作品页审核应滚动到筛选区和第一组作品，确认照片确实出现，而不是只看首屏、标题、按钮或 DOM 数量。
4. **修复后复核指标不完整。** 用户指出问题后，第一反应仍偏向路径解释，没有先用正式域名和本地预览验证客户看到的实际画面。
5. **跨语言共享 CSS 风险没有同步验。** 作品样式由 `public/styles.css` 统一控制，中文、英文、法文、德文作品页都可能受影响，不能只检查单一语言页面是否有图片标签。

#### 正确处理方式

```text
凡是改动 SEO、图片结构、picture/source、懒加载、共享图片样式或 Portfolio 相关 CSS，必须同时验证图片资源加载和真实可见性。作品页至少检查 320px、390px、1440px 三档：作品数量、第一张作品卡片高度、第一张图片高度、横向溢出和真实截图。
```

#### 防复发流程

1. SEO 或图片优化提交后，列出受共享样式影响的页面，尤其是 Portfolio、首页照片流、服务页样片和 About 图片。
2. 对 Portfolio 这类核心作品页，不能只看 `img` 是否存在；必须读取 `getBoundingClientRect()`，确认作品卡片和图片高度都大于 0。
3. 检查 `picture`、`source`、`img`、父级 `figure` 的 `position`、`height`、`aspect-ratio`、`overflow` 是否互相冲突。
4. 用正式域名验证，不只用本地预览；如果用户说线上出错，先打开 `https://www.fionatangstudio.com/` 对应页面确认，不先归因到本地路径。
5. 四语言共用样式时，至少抽查 `/zh/portfolio.html`、`/en/portfolio.html`、`/fr/portfolio.html`、`/de/portfolio.html` 在手机宽度下的第一张作品是否可见。
6. 如果图片使用懒加载，区分“首屏未加载”和“卡片高度为 0”；前者可能正常，后者属于严重显示错误。

#### 输出前自检

- 图片是否不只是加载成功，而是在页面上真实可见？
- 第一张作品卡片和第一张图片的 `width / height` 是否都大于 0？
- 父级是否存在 `overflow: hidden` 裁掉子元素的情况？
- SEO / 图片结构改动是否影响了全站共享图片样式？
- 是否检查了正式域名，而不只是本地预览或 GitHub 默认地址？
- 是否在 320px、390px、1440px 三档确认 Portfolio 图片可见？
- 四语言作品页是否都至少抽查过手机宽度？
